\chapter{Levantamento das Necessidades Funcionais}
\label{chap:necessidades}

Com base no guião do projeto proposto pela coordenação de Engenharia Informática da ESTiG, o objetivo central consiste na criação de um \textit{data center} de serviços de rede gerido de forma estritamente reprodutível, recorrendo exclusivamente a ferramentas \textit{open source}. As necessidades do projeto dividem-se em duas categorias principais: os serviços a disponibilizar (requisitos funcionais) e os requisitos técnicos e de segurança da arquitetura (requisitos não-funcionais).

\section{Serviços de Rede a Disponibilizar}

A infraestrutura contempla um ecossistema de serviços essenciais para uma rede empresarial de pequena ou média dimensão. De forma detalhada, os requisitos funcionais integrados assentam na seguinte especificação:

\begin{itemize}
    \item \textbf{Servidor DNS Recursivo e Autoritativo:} Resolução de nomes para os hosts e serviços da rede interna; filtragem ativa de domínios publicitários e maliciosos; console gráfica de auditoria acessível por HTTP.
    \item \textbf{\textit{Reverse Proxy} HTTPS:} Ponto unificado de entrada na rede para encaminhamento dinâmico de tráfego HTTP/HTTPS com base no subcaminho do pedido e terminação TLS.
    \item \textbf{Servidor Web Estático:} Disponibilização de um painel institucional de estado da infraestrutura através do Nginx com injeção de volumes em modo \textit{read-only} para prevenção de desfiguração (\textit{defacement}).
    \item \textbf{Servidor de Ficheiros Corporativo (Nextcloud):} Mecanismo de armazenamento centralizado em nuvem privada que garante a persistência de dados dos utilizadores de forma isolada, resiliente a falhas e controlada por volumes permanentes.
    \item \textbf{Servidor de E-mail Independente (Docker-Mailserver):} Solução completa de comunicações eletrónicas capaz de gerir caixas de correio locais através dos protocolos padrão SMTP e IMAP, incorporando políticas de segurança nativas (Fail2ban e SpamAssassin).
    \item \textbf{Acesso Remoto Seguro (WireGuard VPN):} Estabelecimento de um túnel cifrado à rede interna a partir de redes exteriores WAN, utilizando obrigatoriamente um mecanismo de \textit{Dynamic DNS} para contornar a volatilidade do IP público do router doméstico.
    \item \textbf{Observabilidade e Logs Centralizados:} Recolha contínua de métricas de telemetria do host, monitorização do ciclo de vida dos contentores e centralização de logs estruturados (Prometheus, Loki, Promtail e Grafana).
\end{itemize}

\section{Requisitos Técnicos e de Arquitetura}

Para além do correto funcionamento dos serviços, a plataforma respeita rigorosos critérios de engenharia e gestão declarativa:
\begin{itemize}
    \item \textbf{Containerização Estruturada:} Todos os serviços correm isolados em \textit{containers} independentes via \texttt{docker-compose.yml}, garantindo portabilidade total.
    \item \textbf{Segurança Ativa (\textit{Hardening}):} Aplicação do princípio do mínimo privilégio em volumes (\texttt{:ro}), isolamento de redes Docker do tipo bridge, cifragem obrigatória de tráfego e proteção criptográfica de segredos com Ansible Vault.
    \item \textbf{\textit{Pipeline} de CI/CD Automation:} Automação baseada em GitLab Runner local para efetuar o deploy controlado da infraestrutura a cada modificação submetida ao repositório Git.
\end{itemize}

A infraestrutura assenta na seguinte segmentação de hardware e virtualização:
\begin{itemize}
    \item \textbf{Virtualização (VMware Workstation):} O hypervisor local VMware Workstation garante estabilidade e controlo granular sobre as placas de rede virtuais \cite{vmware_workstation}. O modo \textit{Bridged} foi selecionado para que a VM interaja diretamente na rede física local (LAN), obtendo o endereço \texttt{192.168.1.252} na mesma gama do router.
    \item \textbf{Sistema Operativo (\textit{Guest OS}):} Optou-se pelo \textbf{Ubuntu Server 24.04 LTS} \cite{ubuntu_server}. Uma distribuição estável sem interface gráfica (\textit{headless}), minimizando o desperdício de recursos computacionais e garantindo compatibilidade nativa com Docker e Ansible.
\end{itemize}

\subsection{Configuração de Rede do Servidor}
A rede do Ubuntu Server é gerida declarativamente pelo \textbf{Netplan} \cite{netplan_docs}. Configurou-se o endereço IP estático (\texttt{192.168.1.252}) no ficheiro \texttt{/etc/netplan/00-installer-config.yaml} para assegurar a perenidade dos serviços de rede locais, aplicando as alterações de forma segura com o comando \texttt{netplan apply}.

\subsection{Acesso Externo: Mapeamento em DMZ e Dynamic DNS (DDNS)}
Para viabilizar o acesso seguro ao túnel VPN WireGuard a partir de redes exteriores WAN, o router residencial foi configurado com uma \textbf{DMZ} (\textit{Demilitarized Zone}) direcionada para o IP estático da VM (\texttt{192.168.1.252}). Desta forma, pedidos externos direcionados à porta UDP 51821 atingem diretamente o contentor WireGuard.

Para contornar o problema dos IPs públicos dinâmicos atribuídos pelos operadores de telecomunicações, que sofrem alterações frequentes e quebrariam as ligações dos clientes, integrou-se um mecanismo de \textbf{Dynamic DNS (DDNS)}. Este mecanismo atualiza automaticamente o nome de domínio fixo \texttt{daniel-infra.ddns.net} com o IP público atual do router doméstico. No parâmetro \texttt{SERVERURL} do WireGuard, em vez de se definir um IP estático volátil, configurou-se o nome de DNS dinâmico registado para este efeito, garantindo conectividade estável e permanente para os utilizadores em mobilidade externa.
