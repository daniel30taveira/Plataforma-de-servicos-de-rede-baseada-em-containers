\chapter{Levantamento das Necessidades Funcionais}
\label{chap:necessidades}

Com base no guião do projeto proposto, o objetivo central consiste na criação de um \textit{data center} de serviços de rede que possa ser gerido de forma reprodutível, recorrendo exclusivamente a ferramentas \textit{open source}. As necessidades do projeto dividem-se em duas categorias principais: os serviços a disponibilizar (requisitos funcionais) e os requisitos técnicos e de arquitetura (requisitos não-funcionais).

\section{Serviços de Rede a Disponibilizar}

A infraestrutura deve contemplar um conjunto de serviços típicos de uma rede de pequena/média dimensão. De forma detalhada, os requisitos funcionais de cada serviço integrado são os seguintes:

\begin{itemize}
    \item \textbf{Servidor DNS:} Resolução de nomes para os hosts e serviços da rede interna; funcionamento em modo recursivo e autoritativo; interface de administração acessível por HTTP.
    \item \textbf{\textit{Reverse Proxy} HTTPS:} Ponto de entrada único na rede para encaminhamento dinâmico de tráfego HTTP/HTTPS com base no caminho do pedido e terminação TLS unificada.
    \item \textbf{Servidor Web Estático:} Disponibilização de uma página institucional de estado da infraestrutura através do Nginx com injeção de volumes em modo \textit{read-only}.
    \item \textbf{Servidor de Ficheiros (Nuvem Privada):} Mecanismo de armazenamento centralizado na rede que garanta a persistência de ficheiros e dados dos utilizadores de forma isolada, resiliente a falhas e controlada por permissões de volume.
    \item \textbf{Servidor de E-mail:} Solução de comunicações eletrónicas interna capaz de gerir caixas de correio locais através dos protocolos padrão SMTP e IMAP, incorporando políticas básicas de segurança contra acessos maliciosos.
    \item \textbf{VPN (WireGuard):} Acesso remoto cifrado e seguro à rede interna de \textit{containers} a partir de dispositivos externos (computador portátil e telemóvel), com suporte a Dynamic DNS para contornar a volatilidade do IP público do router.
    \item \textbf{Monitorização e Logs Centralizados:} Recolha contínua de métricas de \textit{hardware} do host, monitorização do ciclo de vida dos contentores e centralização de logs agregados num único painel analítico para resposta proativa a incidentes.
\end{itemize}

\section{Requisitos Técnicos e de Arquitetura}

Para além do correto funcionamento dos serviços, a plataforma respeita rigorosos critérios de implementação e gestão estabelecidos no guião:
\begin{itemize}
    \item \textbf{Containerização e Configuração Declarativa:} Todos os serviços correm isolados em \textit{containers} via \texttt{docker-compose.yml}, garantindo portabilidade total.
    \item \textbf{Observabilidade e Monitorização:} Integração de telemetria completa (métricas e logs de sistema) e dashboards para alarmística de CPU, RAM e integridade da infraestrutura.
    \item \textbf{Segurança (\textit{Hardening}):} Aplicação do princípio do mínimo privilégio em volumes (\texttt{:ro}), isolamento de redes Docker do tipo bridge, cifragem obrigatória de tráfego e proteção criptográfica de segredos.
    \item \textbf{\textit{Pipeline} de CI/CD:} Automação baseada em GitLab Runner local para efetuar o deploy controlado da infraestrutura a cada modificação submetida ao repositório Git.
\end{itemize}

Para suportar a plataforma e garantir o isolamento do \textit{host} físico, a infraestrutura base foi implementada em ambiente virtualizado.

\begin{itemize}
    \item \textbf{Virtualização (VMware Workstation):} O hypervisor local VMware Workstation garante estabilidade e controlo granular sobre as placas de rede virtuais \cite{vmware_workstation}. O modo \textit{Bridged} foi selecionado para que a VM interaja diretamente na rede física local (LAN), obtendo um IP na mesma gama e tornando-se acessível para os serviços críticos de DNS e VPN.
    \item \textbf{Sistema Operativo (\textit{Guest OS}):} Optou-se pelo \textbf{Ubuntu Server 24.04 LTS} \cite{ubuntu_server}. Uma distribuição sem interface gráfica (\textit{headless}), minimizando o desperdício de recursos computacionais e garantindo compatibilidade alargada com Docker e Ansible.

\subsection{Configuração de Rede do Servidor}
A rede do Ubuntu Server é gerida declarativamente pelo \textbf{Netplan} \cite{netplan_docs}. Configurou-se um endereço IP estático (\texttt{192.168.1.252}) no ficheiro \texttt{/etc/netplan/00-installer-config.yaml} para assegurar a perenidade dos serviços de rede locais, aplicando as alterações de forma segura com o comando \texttt{netplan apply}.

\subsection{Acesso Externo: Mapeamento em DMZ e Dynamic DNS}
Para viabilizar o acesso seguro ao túnel VPN WireGuard a partir de redes externas (ex: dados móveis ou redes públicas), o router residencial foi configurado com uma \textbf{DMZ} (\textit{Demilitarized Zone}) direcionada exclusivamente para o IP estático do servidor (\texttt{192.168.1.252}). Desta forma, pedidos externos não filtrados na porta UDP 51821 atingem diretamente o contentor WireGuard. 

Para contornar o problema dos IPs públicos dinâmicos atribuídos pelos operadores de telecomunicações (MEO), que sofrem alterações frequentes e quebrariam as ligações dos clientes, integrou-se um mecanismo de **Dynamic DNS (DDNS)**. Este atualiza automaticamente um nome de domínio fixo (\texttt{daniel-infra.ddns.net}) com o IP público atual do router, garantindo que os perfis de cliente mantêm a conectividade permanentemente estável sem necessidade de reconfiguração manual.
