\chapter{Arquitetura e Implementação}
\label{chap:implementacao}

Este capítulo detalha o processo prático de implementação da plataforma de serviços de rede, descrevendo não apenas as configurações finais, mas também os problemas reais encontrados e o raciocínio técnico aplicado na sua resolução. A abordagem adotada seguiu as melhores práticas de \textit{Infrastructure as Code} (IaC), garantindo que todo o ambiente pudesse ser reconstruído de forma automatizada e previsível \cite{morris2016infrastructure}.

\section{Arquitetura Geral da Plataforma}

Antes de detalhar a implementação, importa descrever a arquitetura global da plataforma. Esta organiza-se em três camadas hierárquicas com responsabilidades bem definidas:

\begin{enumerate}
    \item \textbf{Camada de infraestrutura:} a máquina física Windows com VMware Workstation a correr uma VM Ubuntu Server 24.04 LTS em modo \textit{Bridged} na rede local.
    \item \textbf{Camada de serviços:} os \textit{containers} Docker (Nginx, Pi-hole, WireGuard, Prometheus, Node Exporter, Grafana) a comunicar através de uma rede interna Docker (\texttt{rede\_interna}).
    \item \textbf{Camada de automação:} o Ansible (executado via WSL na máquina Windows) e a \textit{pipeline} GitLab CI/CD (executada pelo Runner instalado no servidor Ubuntu).
\end{enumerate}

Todo o tráfego externo entra pelo Nginx nas portas 80/443, ou diretamente pelo WireGuard na porta UDP 51821. Os serviços internos comunicam exclusivamente através da rede Docker \texttt{bridge} isolada, sem exposição desnecessária de portas ao exterior.

\subsection{Configuração de IP Estático com Netplan}
\label{sec:netplan}

Para garantir que o servidor mantém sempre o mesmo endereço IP na rede local --- condição essencial para o correto funcionamento do DNS, da VPN e das entradas do inventário Ansible ---, procedeu-se à configuração de um IP estático através do Netplan \cite{netplan_docs}.

\subsubsection*{Problema Encontrado: Erros de Indentação YAML e Falhas de DNS}

A configuração do Netplan revelou-se o primeiro obstáculo técnico significativo do projeto. O ficheiro de configuração \texttt{/etc/netplan/00-installer-config.yaml} utiliza o formato YAML, que é extremamente sensível à indentação: um espaço extra ou a substituição de espaços por tabulações resulta num erro de \textit{parse} que impede a aplicação da configuração.

Durante a configuração, foram cometidos dois erros distintos:

\begin{enumerate}
    \item \textbf{Erro de indentação:} A secção \texttt{nameservers} foi indentada ao nível errado (ao nível de \texttt{ethernets} em vez de ao nível da interface específica), resultando no erro:
    \begin{lstlisting}[language=bash, frame=single, breaklines=true]
Invalid YAML at /etc/netplan/00-installer-config.yaml
    \end{lstlisting}
    O problema foi diagnosticado com o comando \texttt{netplan try}, que aplica a configuração temporariamente durante 120 segundos e reverte automaticamente se não for confirmada --- mecanismo de segurança crucial para evitar perda de acesso remoto ao servidor.

    \item \textbf{Falha de resolução DNS:} Após corrigir a indentação e aplicar o IP estático, o servidor perdeu capacidade de resolução de nomes de domínio. A causa foi a omissão dos servidores \texttt{nameservers} na configuração Netplan. O servidor tentava resolver nomes através de si próprio (o Pi-hole ainda não estava instalado), resultando em falhas de DNS. A solução imediata foi adicionar explicitamente os servidores DNS do Google (\texttt{8.8.8.8} e \texttt{8.8.4.4}) na configuração Netplan, até o Pi-hole estar operacional.
\end{enumerate}

A configuração Netplan final, corretamente indentada, ficou com o seguinte aspeto:

\begin{lstlisting}[language=bash, frame=single, breaklines=true, caption={Configura\c{c}\~ao Netplan para IP est\'atico}]
network:
  version: 2
  ethernets:
    ens33:
      dhcp4: no
      addresses:
        - 192.168.1.252/24
      routes:
        - to: default
          via: 192.168.1.1
      nameservers:
        addresses:
          - 8.8.8.8
          - 8.8.4.4
\end{lstlisting}

\subsection{Problema de \textit{Bridging} no VMware}
\label{sec:vmware_bridging}

Um segundo problema de rede surgiu de forma intermitente após reinícios do \textit{router} da rede doméstica. A VM perdia subitamente a conectividade com a rede física, apesar do Netplan estar corretamente configurado.

\subsubsection*{Diagnóstico}

O diagnóstico começou por verificar o estado da interface de rede no servidor (\texttt{ip addr show ens33} e \texttt{ip route show}), que apresentava a configuração correta. O problema residia, portanto, na camada de virtualização, e não no sistema operativo convidado.

Ao inspecionar o \textbf{VMware Virtual Network Editor} (ferramenta de gestão de redes virtuais do VMware, acessível em \textit{Edit > Virtual Network Editor}), identificou-se a causa: o VMware Workstation mantém uma lista de adaptadores de rede físicos disponíveis para o modo \textit{Bridged}, e após um reinício do \textit{router}, a ferramenta tinha reordenado os adaptadores e selecionado automaticamente um adaptador virtual do Windows (criado pelo \textbf{Hyper-V} ou pelo cliente \textbf{OpenVPN}) em vez da placa de rede física Realtek.

Esta seleção incorreta significa que a VM tentava fazer a ponte de rede (\textit{bridge}) para um adaptador virtual sem ligação física real à rede, tornando-a invisível para os restantes dispositivos da LAN.

\subsubsection*{Solução}

A solução consistiu em forçar manualmente o VMware a utilizar o adaptador correto no Virtual Network Editor:

\begin{enumerate}
    \item Abrir o VMware Virtual Network Editor com privilégios de administrador (\textit{Change Settings}).
    \item Selecionar a rede \texttt{VMnet0} (utilizada pelo modo \textit{Bridged}).
    \item Em vez de \textit{Automatic}, selecionar explicitamente o adaptador físico \textbf{Realtek} (o adaptador Ethernet físico da máquina).
    \item Aplicar as alterações e reiniciar a VM.
\end{enumerate}

\subsection{\textit{Playbook} de Instalação do Docker}

O \textit{Playbook} \texttt{setup\_docker.yml} é responsável por preparar o servidor para receber os \textit{containers}. As suas tarefas são executadas com \texttt{become: yes} (equivalente a \texttt{sudo}), garantindo os privilégios necessários para instalação de pacotes de sistema \cite{ansible_docs}.

As tarefas do \textit{Playbook} são executadas pela seguinte ordem:

\begin{enumerate}
    \item Atualização da \textit{cache} APT (\texttt{apt: update\_cache: yes}).
    \item Instalação de dependências de sistema (\texttt{apt-transport-https}, \texttt{ca-certificates}, \texttt{curl}, \texttt{software-properties-common}).
    \item Instalação do Docker Engine (\texttt{docker.io}) e do \textit{plugin} Docker Compose V2 (\texttt{docker-compose-v2}).
    \item Ativação e início do \textit{daemon} Docker via \texttt{systemd}, com arranque automático no boot.
    \item Adição do utilizador \texttt{daniel} ao grupo \texttt{docker}, implementando o princípio do mínimo privilégio: elimina a necessidade de usar \texttt{sudo} a cada comando Docker.
\end{enumerate}

A execução é iniciada com:

\begin{lstlisting}[language=bash, frame=single, breaklines=true]
ansible-playbook -i inventory.ini setup_docker.yml
\end{lstlisting}

\section{Implementação dos Serviços via Docker Compose}

\subsection{Arquitectura de Rede dos \textit{Containers}}

Todos os serviços são definidos no ficheiro \texttt{docker-compose.yml} e partilham uma rede interna Docker designada \texttt{rede\_interna}, do tipo \texttt{bridge}. Esta rede isola os \textit{containers} da rede do \textit{host} e permite-lhes comunicar entre si através de nomes de serviço (DNS interno do Docker), sem exposição desnecessária de portas ao exterior \cite{docker_security}.

Apenas as portas estritamente necessárias são publicadas no \textit{host}: 80 e 443 (Nginx), 53 TCP/UDP (Pi-hole), 8080 (painel Pi-hole), 9090 (Prometheus), 9100 (Node Exporter), 3000 (Grafana) e 51821 UDP (WireGuard).

\subsection{Servidor DNS: Pi-hole}
\label{sec:pihole}

O Pi-hole é configurado como \textit{container} na rede \texttt{rede\_interna}, expondo as portas 53 TCP e UDP para resolução DNS, e a porta 8080 para o painel de administração web \cite{pihole_docs}.

\subsubsection*{Problema: Palavra-passe não Aplicada pelo Volume Persistente}

Durante a primeira inicialização do Pi-hole, verificou-se que a variável de ambiente \texttt{WEBPASSWORD} definida no \texttt{docker-compose.yml} não estava a ser aplicada: o painel de administração recusava a autenticação com a palavra-passe configurada.

A causa foi identificada após análise dos \textit{logs} do \textit{container} (\texttt{docker logs dns\_pihole}): quando o volume do Pi-hole já existe com configuração persistida de uma inicialização anterior (mesmo que com configurações padrão), a variável \texttt{WEBPASSWORD} é ignorada pelo \textit{entrypoint} da imagem.

A resolução passou por injetar o comando de definição de palavra-passe diretamente no \textit{container} em execução, utilizando \texttt{docker exec}:

\begin{lstlisting}[language=bash, frame=single, breaklines=true, caption={Comando de troubleshooting para definir password no Pi-hole}]
sudo docker exec dns_pihole pihole setpassword 'nova_password'
\end{lstlisting}

Este tipo de intervenção pontual demonstra a flexibilidade das arquiteturas baseadas em \textit{containers}: é possível intervir cirurgicamente dentro do ecossistema do serviço sem comprometer o estado do \textit{host} ou dos restantes serviços. Para evitar recorrência deste problema, a solução definitiva seria eliminar o volume antes de uma nova inicialização limpa: \texttt{docker compose down -v}.

\subsection{Servidor Web e \textit{Reverse Proxy}: Nginx}

O Nginx é configurado com um volume Docker em modo \textit{read-only} (\texttt{:ro}) que mapeia a pasta local \texttt{./web} para o diretório de conteúdo estático do Nginx (\texttt{/usr/share/nginx/html}) \cite{nginx_docs}. Esta abordagem tem dois benefícios:

\begin{enumerate}
    \item \textbf{Desacoplamento de conteúdo e infraestrutura:} o conteúdo HTML pode ser atualizado sem necessidade de reconstruir ou reiniciar o \textit{container}.
    \item \textbf{Segurança por \textit{read-only}:} em conformidade com o princípio do mínimo privilégio \cite{nist_containers}, o \textit{container} Nginx não tem permissões de escrita sobre os ficheiros que serve. Num cenário hipotético de comprometimento do processo Nginx, um atacante não conseguiria modificar os ficheiros de conteúdo.
\end{enumerate}

\subsection{VPN: WireGuard}
\label{sec:wireguard}

O WireGuard foi implementado utilizando a imagem \texttt{lscr.io/linuxserver/wireguard}, mantida pela comunidade LinuxServer.io \cite{wireguard_linuxserver}, que simplifica a geração automática de configurações de cliente relativamente à imagem oficial \cite{donenfeld2017wireguard}. As configurações mais relevantes do serviço são:

\begin{itemize}
    \item \texttt{SERVERURL=2.83.165.235}: IP público do servidor, necessário para que os clientes externos saibam para onde se ligar.
    \item \texttt{SERVERPORT=51821}: porta UDP utilizada pelo WireGuard nesta instalação. A porta padrão (51820) foi alterada para 51821 após se detetar um conflito com outro serviço no \textit{router} MEO, que filtrava especificamente essa porta --- este ajuste foi registado no repositório com a mensagem de \textit{commit} \texttt{fix: muda porta da VPN para 51821 devido a conflito no router} (30 de abril de 2026).
    \item \texttt{PEERS=3}: gera automaticamente três perfis de cliente, incluindo ficheiros de configuração e QR codes. O número de \textit{peers} foi aumentado de 2 para 3 numa iteração posterior (\textit{commit} \texttt{feat: atualiza IP publico e aumenta peers da VPN}), para acomodar um dispositivo adicional.
    \item \texttt{PEERDNS=192.168.1.252}: obriga os clientes VPN a utilizar o Pi-hole como servidor DNS quando ligados, garantindo que o bloqueio de domínios e a resolução de nomes internos funcionam também em modo VPN.
    \item \texttt{ALLOWEDIPS=0.0.0.0/0}: todo o tráfego do cliente é encaminhado pelo túnel VPN (\textit{full tunnel}), garantindo que nenhum tráfego escapa sem encriptação.
\end{itemize}

O \textit{container} WireGuard requer duas \textit{capabilities} adicionais do sistema operativo (\texttt{cap\_add: NET\_ADMIN, SYS\_MODULE}), necessárias para manipular as interfaces de rede do \textit{kernel} e carregar o módulo WireGuard.

\subsubsection*{Problema: Clientes Externos Não Conseguiam Ligar à VPN}
\label{sec:wireguard_debug}

Após configurar o WireGuard e gerar os perfis de cliente, o telemóvel não conseguia estabelecer a ligação VPN a partir de uma rede externa (rede 4G). O indicador de estado da aplicação WireGuard no telemóvel mostrava o estado \textit{handshaking} de forma persistente, sem completar a ligação.

\subsubsection*{Processo de Diagnóstico}

O processo de diagnóstico foi estruturado em camadas, do servidor para o exterior:

\textbf{1. Verificar se o \textit{container} WireGuard está em execução e à escuta:}
\begin{lstlisting}[language=bash, frame=single, breaklines=true]
docker ps | grep wireguard
ss -ulnp | grep 51821
\end{lstlisting}
O resultado confirmou que o \textit{container} estava ativo e a porta 51821 UDP estava à escuta.

\textbf{2. Verificar se os pacotes chegam ao servidor com \texttt{tcpdump}:}

A ferramenta \texttt{tcpdump} permite capturar e analisar o tráfego de rede em tempo real, sendo essencial para distinguir se o problema reside no servidor ou na rede/\textit{router} \cite{rfc8446}. Executou-se o seguinte comando no servidor, enquanto o telemóvel tentava ligar:

\begin{lstlisting}[language=bash, frame=single, breaklines=true, caption={Captura de tr\'afego WireGuard com tcpdump}]
sudo tcpdump -i ens33 udp port 51821 -n
\end{lstlisting}

O resultado foi revelador: \textbf{zero pacotes capturados}. Isto significava que os pacotes enviados pelo telemóvel não estavam a chegar à interface de rede do servidor, eliminando qualquer hipótese de problema na configuração do WireGuard ou do \textit{container}.

\textbf{3. Identificação do problema: router a bloquear tráfego externo:}

A ausência de pacotes na interface \texttt{ens33} indicava que o \textit{router} MEO estava a filtrar o tráfego UDP externo antes de o encaminhar para a rede interna. A solução foi configurar o \textit{router} em modo \textbf{DMZ} apontado para o IP estático do servidor (\texttt{192.168.1.252}), fazendo com que todo o tráfego externo não solicitado seja diretamente reencaminhado para o servidor.

\textbf{Solução aplicada:}
\begin{enumerate}
    \item Aceder à interface de administração do \textit{router} MEO (\texttt{192.168.1.1}).
    \item Ativar o modo DMZ e configurar o IP de destino: \texttt{192.168.1.252}.
    \item Reiniciar fisicamente o \textit{router} (um reinício por software não foi suficiente para aplicar a configuração DMZ).
\end{enumerate}

Após o reinício físico do \textit{router}, repetiu-se o \texttt{tcpdump}: os pacotes UDP do telemóvel passaram a ser visíveis na interface \texttt{ens33}, e a ligação VPN foi estabelecida com sucesso.

\section{Implementação de CI/CD com GitLab}
\label{sec:cicd}

\subsection{Configuração do GitLab Runner}

Para implementar a \textit{pipeline} de CI/CD, foi necessário instalar um \textit{GitLab Runner} no próprio servidor Ubuntu \cite{gitlab_cicd}. A opção por um \textit{runner} local justifica-se pela arquitetura de rede: o servidor encontra-se numa rede privada, inacessível diretamente pelos \textit{runners} partilhados da plataforma GitLab. O modelo \textit{Pull} — onde o \textit{Runner} interpela periodicamente o servidor GitLab para obter novos \textit{jobs} --- resolve este problema sem expor o servidor à Internet.

O \textit{Runner} foi instalado e registado com o executor \texttt{shell}:

\begin{lstlisting}[language=bash, frame=single, breaklines=true, caption={Registo do GitLab Runner}]
sudo gitlab-runner register \
  --url https://gitlab.estig.ipb.pt/ \
  --registration-token <TOKEN> \
  --executor shell \
  --description "runner-datacenter"
\end{lstlisting}

\subsection{Configuração da \textit{Pipeline}: \texttt{.gitlab-ci.yml}}

O ficheiro \texttt{.gitlab-ci.yml} define a \textit{pipeline} de CI/CD. A sua lógica é simples e deliberada: a cada \textit{commit} na \textit{branch} principal, o \textit{Runner} executa o \textit{Playbook} Ansible \texttt{deploy\_servicos.yml} localmente no servidor (usando \texttt{-i "localhost," -c local}), garantindo que os \textit{containers} são atualizados sem intervenção humana.

\begin{lstlisting}[language=bash, frame=single, breaklines=true, caption={Ficheiro .gitlab-ci.yml}]
stages:
  - deploy

deploy_infraestrutura:
  stage: deploy
  tags:
    - meu-runner-local
  script:
    - echo "A iniciar o deploy automatico via CI/CD..."
    - cd infra
    - sudo ansible-playbook -i "localhost," -c local deploy_servicos.yml
    - echo "Deploy concluido com sucesso!"
\end{lstlisting}

Para que o utilizador do \textit{Runner} possa executar o Ansible com \texttt{sudo} sem intervenção manual, foi necessário adicionar uma regra no \texttt{sudoers}:

\begin{lstlisting}[language=bash, frame=single, breaklines=true]
gitlab-runner ALL=(ALL) NOPASSWD: /usr/bin/ansible-playbook
\end{lstlisting}

\section{Observabilidade: Prometheus, Node Exporter e Grafana}

\subsection{Configuração do Prometheus}

O Prometheus é configurado através do ficheiro \texttt{prometheus.yml}, montado como volume \textit{read-only} no \textit{container} \cite{prometheus_docs}. O ficheiro define dois \textit{scrape jobs}:

\begin{itemize}
    \item \texttt{prometheus}: recolhe métricas do próprio Prometheus (porto 9090), permitindo monitorizar o estado do sistema de monitorização.
    \item \texttt{node-exporter}: recolhe métricas de \textit{hardware} do servidor através do Node Exporter (porto 9100). O \textit{target} é referenciado pelo nome do serviço Docker (\texttt{node-exporter:9100}), utilizando o DNS interno da rede Docker.
\end{itemize}

O intervalo de \textit{scrape} foi configurado a 15 segundos, proporcionando granularidade suficiente para detetar anomalias sem sobrecarregar a base de dados.
