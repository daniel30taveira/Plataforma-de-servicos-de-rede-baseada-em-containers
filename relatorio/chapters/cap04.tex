\chapter{Arquitetura e Implementação}
\label{chap:implementacao}
 
Este capítulo detalha o processo prático de implementação da plataforma de serviços de rede, descrevendo não apenas as configurações finais, mas também os problemas reais encontrados e o raciocínio técnico aplicado na sua resolução. A abordagem adotada seguiu as melhores práticas de \textit{Infrastructure as Code} (IaC), garantindo que todo o ambiente pudesse ser reconstruído de forma automatizada e previsível \cite{morris2016infrastructure}.
 
\section{Arquitetura Geral da Plataforma}
 
Antes de detalhar a implementação, importa descrever a arquitetura global da plataforma. Esta organiza-se em três camadas hierárquicas com responsabilidades bem definidas:
 
\begin{enumerate}
    \item \textbf{Camada de infraestrutura:} a máquina física Windows com VMware Workstation a correr uma VM Ubuntu Server 24.04 LTS em modo \textit{Bridged} na rede local.
    \item \textbf{Camada de serviços:} os \textit{containers} Docker (Nginx, Pi-hole, WireGuard, Nextcloud, Docker-Mailserver, Prometheus, Node Exporter, Loki, Promtail e Grafana) a comunicar através de uma rede interna Docker (\texttt{rede\_interna}).
    \item \textbf{Camada de automação:} o Ansible (executado via WSL na máquina Windows) e a \textit{pipeline} GitLab CI/CD (executada pelo Runner instalado no servidor Ubuntu).
\end{enumerate}
 
Todo o tráfego externo entra pelo Nginx nas portas 80/443, ou diretamente pelo WireGuard na porta UDP 51821. Os serviços internos comunicam exclusivamente através da rede Docker \texttt{bridge} isolada, sem exposição desnecessária de portas ao exterior. Para uma melhor compreensão desta segmentação e interligação funcional, a Figura~\ref{fig:esquema_3_camadas} apresenta o esquema conceptual detalhado das três camadas propostas para o sistema.

\subsection{Configuração de IP Estático com Netplan}
\label{sec:netplan}
 
Para garantire que o servidor mantém sempre o mesmo endereço IP na rede local --- condição essencial para o correto funcionamento do DNS, da VPN e das entradas do inventário Ansible ---, procedeu-se à configuração de um IP estático através do Netplan \cite{netplan_docs}.
 
\subsubsection*{Problema Encontrado: Erros de Indentação YAML e Falhas de DNS}
 
A configuração do Netplan revelou-se o primeiro obstáculo técnico significativo do projeto. O ficheiro de configuração \texttt{/etc/netplan/00-installer-config.yaml} utiliza o formato YAML, que é extremamente sensível à indentação: um espaço extra ou a substituição de espaços por tabulações resulta num erro de \textit{parse} que impede a aplicação da configuração.
 
Durante a configuração, foram cometidos dois erros distintos:
 
\begin{enumerate}
    \item \textbf{Erro de indentação:} A secção \texttt{nameservers} foi indentada ao nível errado, resultando no erro \texttt{Invalid YAML at /etc/netplan/00-installer-config.yaml}. O problema foi diagnosticado com o comando \texttt{netplan try}, que aplica a configuração temporariamente durante 120 segundos e reverte automaticamente se não for confirmada --- mecanismo de segurança crucial para evitar perda de acesso remoto ao servidor.
 
    \item \textbf{Falha de resolução DNS:} Após corrigir a indentação e aplicar o IP estático, o servidor ganhou o endereço mas perdeu capacidade de resolução de nomes de domínio. A causa foi a omissão dos \texttt{nameservers} na configuração. O servidor tentava resolver nomes através de si próprio (o Pi-hole ainda não estava instalado), resultando em falhas de DNS. A solução imediata foi adicionar explicitamente os servidores DNS do Google (\texttt{8.8.8.8} e \texttt{8.8.4.4}) na configuração Netplan, até o Pi-hole estar operacional.
\end{enumerate}
 
A configuração Netplan final, corretamente indentada, ficou com o seguinte aspeto:
 
\begin{lstlisting}[language=bash, frame=single, breaklines=true, caption={Configura\c{c}\~ao Netplan para IP est\'atico}]
network:
  version: 2
  ethernets:
    ens33:
      dhcp4: no
      addresses:
        - 192.168.1.252/24
      routes:
        - to: default
          via: 192.168.1.1
      nameservers:
        addresses:
          - 8.8.8.8
          - 8.8.4.4
\end{lstlisting}
 
\subsection{Problema de \textit{Bridging} no VMware}
\label{sec:vmware_bridging}
 
Um segundo problema de rede surgiu de forma intermitente após reinícios do \textit{router} da rede doméstica. A VM perdia subitamente a conectividade com a rede física, apesar do Netplan estar corretamente configurado.
 
O diagnóstico começou por verificar o estado da interface de rede no servidor (\texttt{ip addr show ens33} e \texttt{ip route show}), que apresentava a configuração correta. O problema residia, portanto, na camada de virtualização. Ao inspecionar o \textbf{VMware Virtual Network Editor} (acessível em \textit{Edit > Virtual Network Editor}), identificou-se a causa: após um reinício do \textit{router}, a ferramenta tinha reordenado os adaptadores e selecionado automaticamente um adaptador virtual do Windows (criado pelo \textbf{Hyper-V} ou pelo cliente \textbf{OpenVPN}) em vez da placa de rede física Realtek, tornando a VM invisível na LAN.
 
A solução consistiu em forçar manualmente o VMware a utilizar o adaptador correto: abrir o Virtual Network Editor com privilégios de administrador, selecionar a rede \texttt{VMnet0}, e em vez de \textit{Automatic}, selecionar explicitamente o adaptador físico \textbf{Realtek}. A Figura~\ref{fig:vmware_network_editor} ilustra a configuração correta.
 
\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.85\textwidth]{vmware_network_editor.png}
    \caption{VMware Virtual Network Editor com o adaptador Realtek selecionado manualmente para \texttt{VMnet0} (modo \textit{Bridged}).}
    \label{fig:vmware_network_editor}
\end{figure}
 
Após esta correção, a VM passou a manter a conectividade de rede de forma estável, independentemente de reinícios do \textit{router}.

\subsection{Criptografia e Gestão de Segredos com Ansible Vault}
\label{sec:ansible_vault}
Atendendo à necessidade de robustez na gestão de segredos (\textit{hardening} da plataforma), foi integrada a ferramenta de cifra \textbf{Ansible Vault}. Esta adição arquitetural resolve o risco de manter credenciais administrativas em formato de texto limpo dentro do repositório Git. 

Ficheiros contendo variáveis sensíveis, como a password de administração do Pi-hole (\texttt{WEBPASSWORD}) e do Grafana, foram encriptados de forma simétrica com o algoritmo AES-256. O Ansible descriptografa estas variáveis apenas em memória durante a execução do deploy automático, utilizando um ficheiro de chave temporário e protegido na pipeline, garantindo a integridade e confidencialidade dos segredos do sistema.
 
\section{Implementação dos Serviços via Docker Compose}
 
Todos os serviços são definidos no ficheiro \texttt{docker-compose.yml} e partilham uma rede interna Docker designada \texttt{rede\_interna}, do tipo \texttt{bridge}. Esta rede isola os \textit{containers} da rede do \textit{host} e permite-lhes comunicar entre si através de nomes de serviço (DNS interno do Docker), sem exposição desnecessária de portas ao exterior \cite{docker_security}. Apenas as portas estritamente necessárias são publicadas no \textit{host}: 80 e 443 (Nginx), 53 TCP/UDP (Pi-hole), 8080 (painel Pi-hole), 9090 (Prometheus), 9100 (Node Exporter), 3000 (Grafana), 3100 (Loki) e 51821 UDP (WireGuard).
 
\subsection{Servidor DNS: Pi-hole}
\label{sec:pihole}
 
O Pi-hole é configurado como \textit{container} na rede \texttt{rede\_interna}, expondo as portas 53 TCP e UDP para resolução DNS, e a porta 8080 para o painel de administração web \cite{pihole_docs, pihole_docker}.
 
\subsubsection*{Problema: Palavra-passe não Aplicada pelo Volume Persistente}
 
Durante a primeira inicialização do Pi-hole, verificou-se que a variável de ambiente \texttt{WEBPASSWORD} definida no \texttt{docker-compose.yml} não estava a ser aplicada: o painel de administração recusava a autenticação. A causa foi identificada após análise dos \textit{logs} do \textit{container} (\texttt{docker logs dns\_pihole}): quando o volume do Pi-hole já existe com configuração persistida de uma inicialização anterior, a variável \texttt{WEBPASSWORD} é ignorada pelo \textit{entrypoint} da imagem.
 
A resolução passou por injetar o comando diretamente no \textit{container} em execução:
 
\begin{lstlisting}[language=bash, frame=single, breaklines=true, caption={Comando de troubleshooting para definir password no Pi-hole}]
sudo docker exec dns_pihole pihole setpassword 'nova_password'
\end{lstlisting}
 
A solução definitiva para evitar recorrência seria eliminar o volume antes de uma nova inicialização limpa (\texttt{docker compose down -v}). A Figura~\ref{fig:pihole_admin} comprova o acesso bem-sucedido ao painel de administração do Pi-hole.

    \caption{Interface de administração do servidor DNS (Pi-hole) a correr in \textit{container}.}

\end{figure}
 
\subsection{Servidor Web e \textit{Reverse Proxy}: Nginx}
 
O Nginx é configurado com um volume Docker em modo \textit{read-only} (\texttt{:ro}) que mapeia a pasta local \texttt{./web} para o diretório de conteúdo estático do Nginx (\texttt{/usr/share/nginx/html}) \cite{nginx_docs, nginx_reverseproxy_guide}. Esta abordagem tem dois benefícios: o conteúdo HTML pode ser atualizado sem reconstruir o \textit{container}; e o \textit{container} Nginx não tem permissões de escrita sobre os ficheiros que serve --- em conformidade com o princípio do mínimo privilégio \cite{nist_containers}, um atacante que comprometesse o processo Nginx não conseguiria modificar os ficheiros originais. 

Além disso, o Nginx atua como Reverse Proxy completo, intercetando todos os pedidos HTTP direcionados para os subcaminhos organizados da plataforma (como \texttt{/grafana/} e \texttt{/nextcloud/}), encaminhando o tráfego de forma limpa pela rede interna sem expor os portos das aplicações para o exterior. A Figura~\ref{fig:pagina_web_nginx} demonstra a página de estado da infraestrutura.

\subsection{VPN: WireGuard}
\label{sec:wireguard}
 
O WireGuard foi implementado utilizando a imagem \texttt{lscr.io/linuxserver/wireguard}, mantida pela comunidade LinuxServer.io \cite{wireguard_linuxserver}, que simplifica a geração automática de configurações de cliente relativamente à imagem oficial \cite{donenfeld2017wireguard}. As configurações mais relevantes do serviço são:
 
\begin{itemize}
    \item \texttt{SERVERURL=daniel-infra.ddns.net}: Utilização obrigatória de um nome de \textbf{Dynamic DNS (DDNS)} em vez de um endereço IP público estático. Esta abordagem resolve a volatilidade do IP público atribuído pelo operador residencial, garantindo que mesmo que o IP mude, o domínio aponta automaticamente para o novo endereço e os clientes mantêm a capacidade de estabelecer o túnel de forma persistente.
    \item \texttt{SERVERPORT=51821}: porta UDP utilizada pelo WireGuard nesta instalação. A porta padrão (51820) foi alterada para 51821 após se detetar um conflito com outro serviço no \textit{router} MEO --- este ajuste foi registado no repositório com a mensagem de \textit{commit} \texttt{fix: muda porta da VPN para 51821 devido a conflito no router} (30 de abril de 2026).
    \item \texttt{PEERS=3}: gera automaticamente três perfis de cliente, incluindo ficheiros de configuração e QR codes. O número de \textit{peers} foi aumentado de 2 para 3 numa iteração posterior (\textit{commit} \texttt{feat: atualiza IP publico e aumenta peers da VPN}), para acomodar um dispositivo adicional.
    \item \texttt{PEERDNS=192.168.1.252}: obriga os clientes VPN a utilizar o Pi-hole como servidor DNS, garantindo que o bloqueio de domínios e a resolução de nomes internos funcionam também em modo VPN.
    \item \texttt{ALLOWEDIPS=0.0.0.0/0}: encaminha todo o tráfego do cliente pelo túnel VPN (\textit{full tunnel}).
\end{itemize}
 
O \textit{container} WireGuard requer duas \textit{capabilities} adicionais do sistema operativo (\texttt{cap\_add: NET\_ADMIN, SYS\_MODULE}), necessárias para manipular as interfaces de rede do \textit{kernel}. A Figura~\ref{fig:wireguard_qr} mostra o QR code gerado automaticamente pelo \textit{container} para configuração do cliente móvel.
 
\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.5\textwidth]{wireguard_qr.png}
    \caption{QR code gerado pelo \textit{container} WireGuard para configuração automática do cliente móvel (\textit{peer1}).}
    \label{fig:wireguard_qr}
\end{figure}
 
\subsubsection*{Problema: Clientes Externos Não Conseguiam Ligar à VPN}
\label{sec:wireguard_debug}
 
Após configurar o WireGuard e gerar os perfis de cliente, o telemóvel não conseguia estabelecer a ligação VPN a partir de uma rede externa (rede 4G), permanecendo no estado \textit{handshaking} de forma persistente.
 
O processo de diagnóstico foi estruturado em camadas. Primeiro, verificou-se que o \textit{container} estava ativo e a porta 51821 UDP estava à escuta (\texttt{ss -ulnp | grep 51821}). Depois, utilizou-se o \texttt{tcpdump} para capturar o tráfego na interface de rede do servidor enquanto o telemóvel tentava ligar:
 
\begin{lstlisting}[language=bash, frame=single, breaklines=true, caption={Captura de tr\'afego WireGuard com tcpdump}]
sudo tcpdump -i ens33 udp port 51821 -n
\end{lstlisting}
 
O resultado foi revelador: \textbf{zero pacotes capturados}. Isto significava que os pacotes enviados pelo telemóvel não estavam a chegar à interface de rede do servidor, eliminando qualquer hipótese de problema na configuração do WireGuard ou do \textit{container}. A ausencia de pacotes na interface \texttt{ens33} indicava que o \textit{router} MEO estava a filtrar o tráfego UDP externo.
 
A solução foi configurar o \textit{router} em modo \textbf{DMZ} \cite{dmz_cloudflare} apontado para o IP do servidor (\texttt{192.168.1.252}), seguido de um reinício físico do \textit{router} (um reinício por software não foi suficiente). Após isso, os pacotes UDP passaram a ser visíveis no \texttt{tcpdump} e a ligação VPN foi estabelecida com sucesso, conforme comprova a Figura~\ref{fig:wireguard_ligado}.
 
\begin{figure}[htbp]

\section{Implementação de CI/CD com GitLab}
\label{sec:cicd}
 
Para implementar a \textit{pipeline} de CI/CD, foi necessário instalar um \textit{GitLab Runner} no próprio servidor Ubuntu \cite{gitlab_cicd, gitlab_runner_docs}. A opção por um \textit{runner} local justifica-se pela arquitetura de rede: o servidor encontra-se numa rede privada, inacessível diretamente pelos \textit{runners} partilhados da plataforma GitLab. O modelo \textit{Pull} --- onde o \textit{Runner} interpela periodicamente o servidor GitLab para obter novos \textit{jobs} --- resolve este problema sem expor o servidor à Internet. O \textit{Runner} foi registado com o executor \texttt{shell} e ficou imediatamente visível e ativo na interface web do GitLab, como comprovam as Figuras~\ref{fig:runner_registo} e~\ref{fig:runner_online}.

O ficheiro \texttt{.gitlab-ci.yml} define a \textit{pipeline} de CI/CD. A cada \textit{commit} na \textit{branch} principal, o \textit{Runner} executa o \textit{Playbook} Ansible \texttt{deploy\_servicos.yml} localmente no servidor, garantindo que os \textit{containers} são atualizados sem intervenção humana \cite{cicd_youtube}:
 
\begin{lstlisting}[language=bash, frame=single, breaklines=true, caption={Ficheiro .gitlab-ci.yml}]
stages:
  - deploy
 
deploy_infraestrutura:
  stage: deploy
  tags:
    - meu-runner-local
  script:
    - echo "A iniciar o deploy automatico via CI/CD..."
    - cd infra
    - sudo ansible-playbook -i "localhost," -c local deploy_servicos.yml
    - echo "Deploy concluido com sucesso!"
\end{lstlisting}
 
Para que o utilizador do \textit{Runner} possa executar o Ansible com \texttt{sudo} sem intervenção manual, foi necessário adicionar uma regra no \texttt{sudoers}: \texttt{gitlab-runner ALL=(ALL) NOPASSWD: /usr/bin/ansible-playbook}.
