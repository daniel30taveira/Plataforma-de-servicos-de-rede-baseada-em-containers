\chapter{Testes e Avaliação}
\label{cap:test}

Este capítulo apresenta os testes realizados para verificar que a plataforma implementada cumpre os objetivos definidos no Capítulo \ref{chap:necessidades}. Para cada serviço, descreve-se o método de teste, o resultado esperado e o resultado obtido. Complementarmente, discutem-se os resultados de forma crítica, identificando o que funcionou conforme esperado, o que exigiu ajustamentos e o que permanece como trabalho futuro.

\section{Metodologia de Testes}

Os testes foram estruturados em dois níveis:

\begin{itemize}
    \item \textbf{Testes de funcionamento individual (\textit{smoke tests}):} verificação de que cada serviço arranca corretamente, responde nas portas esperadas e realiza a sua função básica.
    \item \textbf{Testes de integração:} verificação de que os serviços interagem corretamente entre si (por exemplo, que a VPN utiliza o Pi-hole como DNS, e que o Prometheus recolhe métricas do Node Exporter).
\end{itemize}

\section{Teste 1: Servidor DNS (Pi-hole)}

\textbf{Descrição:} Verificar se o Pi-hole responde a pedidos DNS na porta 53 e resolve corretamente nomes de domínio externos e internos.

\textbf{Resultado esperado:} Resolução bem-sucedida de \texttt{google.com} e latência inferior à do DNS do \textit{router} (graças ao \textit{cache} local).

\textbf{Procedimento:}
\begin{lstlisting}[language=bash, frame=single, breaklines=true]
# Test DNS resolution from another device on the network
nslookup google.com 192.168.1.252
dig @192.168.1.252 google.com

# Check container status and logs
docker logs dns_pihole --tail 20
\end{lstlisting}

\textbf{Resultado obtido:} O Pi-hole respondeu corretamente. O painel de administração web (acessível em \texttt{http://192.168.1.252:8080/admin}) confirmou os pedidos DNS a ser processados e registados em tempo real.

\textbf{Discussão:} O problema de palavra-passe descrito na Secção \ref{sec:pihole} foi o único obstáculo encontrado neste serviço. Após a sua resolução, o Pi-hole manteve-se estável. Uma melhoria futura seria configurar entradas DNS locais (registos A personalizados) para os serviços internos, permitindo acesso por nome (e.g., \texttt{grafana.datacenter.local}).

\section{Teste 2: Servidor Web (Nginx)}

\textbf{Descrição:} Verificar se a página HTML estática é servida corretamente pelo Nginx através do volume \textit{read-only}.

\textbf{Resultado esperado:} Acesso à página de estado da infraestrutura a partir de qualquer dispositivo da rede local.

\textbf{Procedimento:}
\begin{lstlisting}[language=bash, frame=single, breaklines=true]
# Test with curl
curl -I http://192.168.1.252

# Check response headers
curl -v http://192.168.1.252 2>&1 | grep "< HTTP"
\end{lstlisting}

\textbf{Resultado obtido:} O Nginx respondeu com HTTP 200 OK. A página foi acessível a partir de um browser na rede local, confirmando o correto mapeamento do volume.

\textbf{Verificação do modo read-only:}
\begin{lstlisting}[language=bash, frame=single, breaklines=true]
# Tentar escrever dentro do container (deve falhar)
docker exec nginx_proxy touch /usr/share/nginx/html/teste.txt
# Resultado esperado: touch: cannot touch '...': Read-only file system
\end{lstlisting}

\textbf{Resultado obtido:} O sistema de ficheiros dentro do \textit{container} é efetivamente \textit{read-only}, confirmando a aplicação correta da medida de segurança.

\section{Teste 3: VPN (WireGuard)}

\textbf{Descrição:} Verificar se um cliente externo (telemóvel em rede 4G) consegue estabelecer a ligação VPN e aceder aos serviços da rede interna.

\textbf{Resultado esperado:} Ligação VPN estabelecida, acesso ao painel Pi-hole e ao Grafana a partir da rede 4G, com todo o tráfego encriptado.

\textbf{Procedimento e Resultado:} O processo de diagnóstico e resolução do problema do WireGuard foi descrito em detalhe na Secção \ref{sec:wireguard_debug}. Após ativação do modo DMZ no \textit{router} e reinício físico, a ligação VPN foi estabelecida com sucesso.

\textbf{Verificação da encriptação e do roteamento DNS:}

Com a VPN ativa no telemóvel, verificou-se:
\begin{itemize}
    \item O endereço IP público mostrado por \texttt{https://ip.me} mudou para o IP do servidor (confirmando que o tráfego passa pelo túnel).
    \item A resolução DNS passou a ser feita pelo Pi-hole (\texttt{PEERDNS=192.168.1.252}), visível no painel de consultas DNS do Pi-hole.
    \item Os serviços internos (Grafana em \texttt{192.168.1.252:3000}) ficaram acessíveis a partir do telemóvel em rede 4G.
\end{itemize}

\section{Teste 4: \textit{Pipeline} CI/CD}

\textbf{Descrição:} Verificar se uma alteração ao repositório (commit) desencadeia automaticamente o re-\textit{deploy} da infraestrutura sem intervenção humana.

\textbf{Resultado esperado:} \textit{Pipeline} executada com sucesso (estado \textit{passed}) após cada \textit{commit} na \textit{branch} principal.

\textbf{Procedimento:}
\begin{enumerate}
    \item Alterar o ficheiro \texttt{web/index.html} (modificar o texto de estado).
    \item Fazer \textit{commit} e \textit{push} para a \textit{branch} principal.
    \item Observar a \textit{pipeline} na interface GitLab.
\end{enumerate}

\textbf{Resultado obtido:} A \textit{pipeline} foi automaticamente acionada, o \textit{Playbook} Ansible executado e o \textit{container} Nginx atualizado. A página web no browser refletiu a alteração após o \textit{deploy}. A Figura \ref{fig:pipeline_sucesso} comprova o estado final da \textit{pipeline}.

\section{Teste 5: Monitorização (Prometheus + Grafana)}

\textbf{Descrição:} Verificar se o Prometheus recolhe métricas do Node Exporter e se o Grafana as visualiza corretamente.

\textbf{Resultado esperado:} \textit{Targets} Prometheus no estado \textit{UP}; painel Grafana a exibir métricas de CPU, RAM e rede em tempo real.

\textbf{Procedimento:}
\begin{lstlisting}[language=bash, frame=single, breaklines=true]
# Verificar targets do Prometheus
curl http://192.168.1.252:9090/api/v1/targets | python3 -m json.tool | grep health
\end{lstlisting}

\textbf{Resultado obtido:} Ambos os \textit{targets} (\texttt{prometheus} e \texttt{node-exporter}) apresentaram o estado \texttt{"health":"up"}. O painel \textit{Node Exporter Full} no Grafana exibiu corretamente os gráficos de utilização de CPU, memória RAM disponível, I/O de disco e tráfego de rede.

\section{Análise Crítica dos Resultados}

\subsection{Objetivos Cumpridos}

\begin{itemize}
    \item Todos os serviços definidos (DNS, Web, VPN, Monitorização) foram implementados e validados.
    \item A infraestrutura é totalmente reprodutível a partir do repositório Git.
    \item A \textit{pipeline} CI/CD automatiza o \textit{deploy} com sucesso.
    \item As medidas de segurança (\textit{read-only}, isolamento de rede, WireGuard) foram aplicadas e verificadas.
\end{itemize}

\subsection{O que Poderia ter sido Feito de Forma Diferente}

\begin{itemize}
    \item \textbf{Gestão de segredos:} As palavras-passe (Pi-hole, Grafana) estão definidas em texto simples no \texttt{docker-compose.yml}, o que não é adequado para ambientes de produção. Uma melhoria seria utilizar o \textbf{Ansible Vault} para cifrar os segredos no repositório, ou um sistema dedicado como o \textbf{HashiCorp Vault}.
    \item \textbf{HTTPS com certificado real:} O Nginx está configurado para HTTP na porta 80. A implementação de HTTPS com certificados \textit{Let's Encrypt} (via \textbf{Certbot} ou \textbf{Traefik}) seria o próximo passo natural para um ambiente de produção \cite{rfc8446}.
    \item \textbf{Centralização de \textit{logs}:} A \textit{stack} atual não inclui centralização de \textit{logs} dos \textit{containers}. A adição do \textbf{Loki} (ou da \textit{stack} ELK) permitiria pesquisa e correlação de eventos entre serviços.
\end{itemize}

\subsection{Objetivos que Ficaram por Cumprir}

\begin{itemize}
    \item \textbf{Servidor de E-mail:} Não foi implementado um servidor de e-mail. A complexidade de configuração de um servidor SMTP/IMAP seguro (reputação de IP, registos SPF/DKIM/DMARC, gestão de certificados) excedeu o âmbito temporal do projeto. Uma opção futura seria o \textbf{Mailcow} ou o \textbf{docker-mailserver}.
    \item \textbf{Servidor de Ficheiros:} Não foi implementado um servidor de ficheiros (e.g., \textbf{Nextcloud} ou \textbf{Samba}). Este serviço ficou como trabalho futuro.
    \item \textbf{Alertas automáticos no Grafana:} Foram configurados painéis de visualização, mas não regras de alerta (e.g., notificação por e-mail quando a utilização de CPU ultrapassa 80\%).
\end{itemize}
